🔌 IPC & Socket Programming

Inter-Process Communication Concepts

Inter-Process Communication (IPC) adalah mekanisme untuk proses-proses yang berjalan pada sistem operasi yang sama untuk berkomunikasi dan bertukar data. IPC fundamental untuk membangun sistem terdistribusi yang efisien.

Pipes (Pipa)

  • • Unnamed pipes untuk parent-child process
  • • Named pipes untuk arbitrary processes
  • • FIFO data structure
  • • Satu arah komunikasi

Message Queues

  • • POSIX message queues
  • • Asynchronous komunikasi
  • • Prioritas message support
  • • Bidirectional communication

Shared Memory

  • • Fast direct memory access
  • • Memerlukan synchronization
  • • Mutex dan semaphore
  • • Low latency communication

Signals

  • • Asynchronous notifications
  • • Signal handling mechanism
  • • Simple komunikasi
  • • Event-driven architecture
// Socket Programming Example (TCP Client-Server) #include <stdio.h> #include <stdlib.h> #include <unistd.h> #include <sys/socket.h> #include <netinet/in.h> #include <string.h> // Server Socket Creation int server_socket = socket(AF_INET, SOCK_STREAM, 0); struct sockaddr_in server_addr; server_addr.sin_family = AF_INET; server_addr.sin_port = htons(PORT); server_addr.sin_addr.s_addr = INADDR_ANY; bind(server_socket, (struct sockaddr *)&server_addr, sizeof(server_addr)); listen(server_socket, 5); // Client Socket Creation int client_socket = socket(AF_INET, SOCK_STREAM, 0); connect(client_socket, (struct sockaddr *)&server_addr, sizeof(server_addr));

📞 Remote Procedure Calls

RPC Fundamentals

Remote Procedure Call (RPC) adalah protokol yang memungkinkan program untuk memanggil fungsi pada mesin remote seolah-olah memanggil lokal. RPC abstraksi networking complexity dan menyediakan transparent remote execution.

Client Process
1. Call RPC Stub
2. Marshal Arguments
3. Send Request
Network
Data Transmission
Protocol
Server Process
1. Receive Request
2. Unmarshal Args
3. Execute Procedure

📨 Indirect Communication

Message-Oriented Middleware

Indirect communication memisahkan sender dan receiver secara temporal dan spatial. Sistem tidak memerlukan knowledge langsung tentang masing-masing.

Publish-Subscribe Pattern

Publisher

Mengirim message ke topic tanpa mengetahui subscribers

Subscriber

Menerima messages dari topic yang disubscribe

Message Broker

Mengelola distribution dan routing messages

Queue Pattern

Producer

Mengirim message ke queue

Consumer

Mengkonsumsi message dari queue

Queue Manager

FIFO ordering dan delivery guarantee

// RabbitMQ Publisher-Subscriber Example import pika connection = pika.BlockingConnection(pika.ConnectionParameters('localhost')) channel = connection.channel() # Declare exchange channel.exchange_declare(exchange='logs', exchange_type='fanout') # Publish message channel.basic_publish(exchange='logs', routing_key='', body='System message') # Consumer def callback(ch, method, properties, body): print(f"Received message: {body}") result = channel.queue_declare(queue='', exclusive=True) queue_name = result.method.queue channel.queue_bind(exchange='logs', queue=queue_name) channel.basic_consume(queue=queue_name, on_message_callback=callback, auto_ack=True)

🔄 Replication & Consistency

Replication Strategies

Master-Slave Replication

Master

Menerima write operations

Slave

Replicate data dari master

Kelebihan

Simple implementation, read scaling

Kekurangan

Single point of failure, replication lag

Multi-Master Replication

Multiple Masters

Semua nodes bisa handle writes

Synchronization

Conflict resolution mechanism

Kelebihan

High availability, no single point of failure

Kekurangan

Conflict handling complexity

Consistency Models

Model Description Use Case
Strong Consistency Immediate update visibility setelah write Banking systems, financial transactions
Eventual Consistency Delayed consistency, eventually consistent state Social media, DNS systems
Causal Consistency Preserve causal relationships antar operations Distributed databases, message systems
Read-Your-Writes Client melihat own writes immediately Session-based applications

🤝 Consensus Algorithms

Distributed Consensus

Consensus algorithms memastikan semua nodes dalam distributed system mencapai agreement tentang state meskipun menghadapi network failures atau byzantine faults.

Raft

  • • Leader-based consensus
  • • Log replication
  • • Understandable algorithm
  • • Used: etcd, Consul

Paxos

  • • Classic algorithm
  • • Byzantine-tolerant
  • • Complex implementation
  • • Used: Google Chubby

PBFT

  • • Byzantine Fault Tolerant
  • • Practical implementation
  • • Async operations
  • • Used: Blockchain systems

Raft Algorithm Flow

1. Leader Election
Followers mengirim RequestVote RPCs ke candidates selama timeout period
2. Log Replication
Leader menerima client requests dan replicate log entries ke followers
3. Safety
Memastikan log konsistency dan state machine safety
// Raft Consensus Pseudocode class RaftNode { private int currentTerm; private int votedFor; private LogEntry[] log; void AppendEntries(term, leaderId, prevLogIndex, prevLogTerm, entries[], leaderCommit) { if (term < currentTerm) return false; if (prevLogIndex > 0 && log[prevLogIndex].term != prevLogTerm) { return false; } // Append entries dan update commitIndex for (LogEntry entry : entries) { log.append(entry); } if (leaderCommit > commitIndex) { commitIndex = min(leaderCommit, log.length - 1); } return true; } }
